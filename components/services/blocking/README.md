
# Документация проекта: HotKey Blocker & Key Logger (`blocking.exe`)
## 🛠 1. Общее описание системы
Программа представляет собой низкоуровневую системную утилиту для Windows, выполняющую две основные задачи:
1. **Перехват и логирование (Key Logging)**: Отслеживание физических и программных нажатий клавиш клавиатуры, определение текущей языковой раскладки (включая её HKL дескриптор) и вывод подробных метаданных о событии в реальном времени в формате JSON.
2. **Блокировка комбинаций (HotKey Blocking)**: Фильтрация вводимых пользователем клавиш и их сочетаний на основе аргументов командной строки. Если комбинация заблокирована, она перехватывается хуком и «гасится» (не доходит до операционной системы).

## 📂 2. Архитектура и разбиение по файлам

Проект спроектирован по модульному принципу объектно-ориентированного программирования (ООП), исключая хаотичное глобальное состояние и избыточные вызовы системного API.

```text
KeyBlocker/
├── AppConfig.h          # Объединенные структуры конфигурации и данных
├── KeyDictionary.h     # Класс-справочник виртуальных кодов (VK_CODE)
├── KeyDictionary.cpp
├── HotkeyManager.h      # Менеджер комбинаций, перестановок и правил блокировки
├── HotkeyManager.cpp
├── LanguageMonitor.h    # Поток отслеживания раскладок и работы с Pipe
├── LanguageMonitor.cpp
├── KeyboardHook.h       # Низкоуровневый хук WH_KEYBOARD_LL и генератор JSON
├── KeyboardHook.cpp
└── main.cpp             # Точка входа, менеджмент потоков и инициализация
```

## 📄 3. Описание модулей и классов

### 🔹 3.1. Данные: `AppConfig.h`
Содержит структуры, заменяющие разрозненные глобальные переменные.
* `Hotkey`: Хранит текстовую комбинацию (например, `ctrl+alt+t`) и её статус.
* `KeyStateInfo`: Объект-контейнер для сбора всех метаданных о нажатии (код, имя, длительность, статус блокировки, имя панели, текущая раскладка).

### 🔹 3.2. Словарь клавиш: `KeyDictionary`
Инвариантный класс для работы с базой данных клавиш.
* **Назначение**: Читает конфигурационный файл `key_code.txt`, сопоставляет hex-коды клавиш (например, `0x11`) с их строковыми именами (например, `_left_ctrl`).
* **Оптимизация**: Использование `std::unordered_map` обеспечивает константную скорость поиска O(1) при обработке высокоинтенсивного ввода.

### 🔹 3.3. Логика комбинаций: `HotkeyManager`
Ядро анализа ввода пользователя и вычисления модификаторов.
* **Генерация перестановок (`generatePermutations`)**: Рекурсивный алгоритм, вычисляющий все возможные варианты написания горячей клавиши (если пользователь передал `ctrl+alt+t`, менеджер автоматически заблокирует и `alt+ctrl+t`, `t+ctrl+alt` и т.д.).
* **Анализ модификаторов (`processKeyName`)**: Выделяет из общей массы зажатых клавиш управляющие модификаторы (Shift `=`, Ctrl `_`, Alt `-`), очищает их имена от префиксов разметки и формирует массив возможных комбинаций для текущего хука.

### 🔹 3.4. Мониторинг языка: `LanguageMonitor`
Обеспечивает синхронизацию языковой раскладки ввода.
* **Асинхронный пайп (`startPipeListener`)**: Работает в отдельном изолированном потоке через Named Pipe `\\.\pipe\LangHookPipe`. Принимает JSON-подобные строки с метаданными раскладки от внешней библиотеки инъекции (`g.dll`).
* **Потокобезопасность**: Запись и чтение строки `_currentInfo` защищены мьютексом `std::mutex`, исключая состояние гонки (Race Condition) с потоком низкоуровневого хука.

### 🔹 3.5. Перехватчик: `KeyboardHook`
Интерфейс обратного вызова (Callback) для Windows.
* **Обработка событий (`hookCallback`)**: Вызывается операционной системой на каждое нажатие/отпускание. Рассчитывает время удержания клавиши (`duration`), определяет целевую панель устройства (`Main` или `NumPad`), учитывая состояние флага `LLKHF_EXTENDED` и статус `NumLock`.
* **Трансляция символов (`ToUnicodeEx`)**: Использует извлеченный из пайпа дескриптор `HKL` для корректной генерации регистра и символа клавиши (`registerKey`), учитывая CapsLock и Shift.
* **Вывод данных (`printKeyInfo`)**: Формирует валидную JSON-строку. Использование `std::endl` вместо `\n` гарантирует мгновенный сброс буфера `stdout` (реалтайм вывод без залипаний).

## 📡 4. Спецификация вывода (Формат JSON)

Каждое событие клавиатуры транслируется в поток `stdout` в виде одной JSON-строки.

### Пример выходных данных:
```json
{
  "key_code": "0xA2",
  "key_name": "left_ctrl",
  "key": "",
  "status": "Down",
  "duretion": "0",
  "numpan": "Main",
  "pressed_keys": "left_ctrl",
  "option": ["left_ctrl"],
  "blocked": "No blocked",
  "isInjected": "Physical",
  "layout": {"Name": "English", "HKL": "0x0000000004090409", "ID": "0x0409"}
}
```

### Описание полей:

| Поле | Тип | Описание |
| :--- | :--- | :--- |
| `key_code` | `string` | Шестнадцатеричный код виртуальной клавиши (Virtual-Key Code) |
| `key_name` | `string` | Текстовое имя клавиши из словаря, очищенное от префиксов |
| `key` | `string` | Результирующий печатный символ с учетом раскладки и регистра |
| `status` | `string` | Состояние клавиши (`Down` — нажата, `Up` — отпущена) |
| `duretion` | `string` | Время удержания клавиши в миллисекундах (сохранена исходная опечатка) |
| `numpan` | `string` | Панель размещения клавиши (`Main` — основная, `NumPad` — цифровая) |
| `pressed_keys` | `string` | Текущая зафиксированная заблокированная комбинация |
| `option` | `array` | Массив всех возможных комбинаций с учетом модификаторов |
| `blocked` | `string` | Статус перехвата (`Blocked` — ввод заблокирован, `No blocked` — пропущен в ОС) |
| `isInjected` | `string` | Источник ввода (`Physical` — физическая клавиатура, `Programm` — симуляция софтом) |
| `layout` | `object` | Сложный объект с информацией о текущем языке ввода системы |

## ⚡ 5. Ключевые оптимизации и исправления

1. **Устранение задержек (Реалтайм)**: Отключена полная блочная буферизация `setvbuf`. Замена `\n` на `std::endl` обеспечила немедленный вывод JSON при одиночных кликах.
2. **Исправление логики префиксов**: Метод `processKeyModifier` корректно модифицирует строку по ссылке, удаляя префиксы `_`, `=`, `-` на раннем этапе. Это предотвращает появление служебных символов в поле `key_name` и нормализует детекцию.
3. **Стандарт C++20**: Проект переведен на C++20 для легитимного использования назначенных инициализаторов (Designated Initializers) в структурах Windows API (`PROCESSENTRY32W`).
4. **Ресурсосбережение (CPU)**: В бесконечный цикл функции `TerminateOtherInstances` (убийство конкурирующих процессов `blocking.exe`) добавлена задержка `Sleep/std::this_thread::sleep_for`, что снизило утилизацию процессора этим потоком с 15-20% до ~0%.

## 🚀 6. Порядок развертывания и запуска

1. **Компиляция**: Сборка осуществляется через CMake в режиме конфигурации `Release` для применения оптимизаций компилятора MSVC по скорости (`/O2`).
2. **Окружение**: Исполняемый файл `blocking.exe` должен находиться в одной директории с `key_code.txt` и `g.dll`.
3. **Интерфейс запуска**:
   ```cmd
   blocking.exe ctrl+alt+delete shift+q
   ```
